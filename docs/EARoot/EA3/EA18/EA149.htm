Multiplicity indicates to what extent the data might logically exist, independent of the data being known. For example, every physical entity (within the scope of ITS - near the face of the earth) has a geoPosition, but this does not mean that the geoPosition is known.<br/><br/>Rules to consider for modelling<br/><ol>
<li>All objects (instances of a class) should be uniquely identifiable  (but we don't have to have attributes for the keys if they are provided by context; for example PhysicalEntity.geoPosition does not need GeoLocation.id as the context already says there is only one geoPosition)</li><li>Values should have an unambiguous meaning and the meaning should have an unambiguous value (e.g., there should not be two ways to represent the same value)</li><li>All attributes and association role names should be unique within their scope</li><li>The layout of classes, attributes, and associations do not matter</li><li>All attributes should be single valued (1NF) (multiplicity of 1 or 0..1)</li><li>Attribute values should not be shared across objects (2NF)  (i.e., if an attribute value changes for one object; it should not impact the value of the same attribute in other objects)</li><li>Attributes should not have dependencies with non-identifying attributes (3NF)  (e.g., the value and accuracy of speed should be a separate class rather than both being attributes of vehicle)</li><li>Attributes should have</li></ol>
a) Name<br/> b) Definition<br/> c) Multiplicity of 0..1 if the data might not logically exist<br/> d) Suggested datatype to promote reuse as follows<br/> i) If a code; the enumeration <br/> ii) Units (e.g., SI and order of magnitude)<br/> iii) Guidance on resolutions used<br/> e) Traceability information<br/><ol>
<li>Navigable associations should have</li></ol>
a) Role name <br/> b) Definition<br/> c) Multiplicity <br/> d) Qualifiers when necessary to disambiguate multiples<br/>10. Version information is handled via GitHub and approval process <br/><br/>
